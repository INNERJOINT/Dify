# P3：自然语言增强方案 — 评审与改进建议

- 日期：2026-03-19
- 针对文档：`2026-03-19-p3-natural-language-enhancement-plan.md`

## 整体评价

这份详细实施方案整体逻辑清晰、技术选型务实。特别是**不盲目引入向量数据库**，并在第一版优先寻求**轻量化和低耦合**的架构设计非常值得肯定。不过，这套方案在一些边界条件、工程实现细节以及特定组件的匹配度上存在几个关键问题和隐患。

以下是具体的评审分析和改进建议。

---

## 1. 核心意图分类逻辑存在截断漏洞 (对应 Section 2)

**问题描述：**
方案中 2.1 节的 `classify_query` 采用“规则优先”策略，这是一个很大的隐患。正则陷阱与重叠特征会导致误判：目前的正则表达式 `re.search(r'[A-Z][a-z]+[A-Z]', query)` 以及强制短路返回 `exact`，会导致许多混合型自然语言查询被误判。
*   例如用户搜索：“**SystemServer 的初始化过程**”。由于包含了 `SystemServer`，瞬间命中了正则，分类器会返回 `exact`。此时由于后半部分是自然语言句子，这串查询会原封不动地交给 Zoekt 精确搜索，结果必然是 **0 召回**。
*   此外，类似 `getService` 这种首字母不小写的小驼峰命名，根本无法命中大驼峰正则，从而掉进默认的自然语言分支。

**改进建议：Zero-Result Fallback (零结果降级)**
在代码搜索场景，与其在前端做重型或者易碰撞的分类匹配规则，不如**默认所有包含代码倾向特征的 query 都先打给 Zoekt**（Zoekt 极快，一般在 20-50ms 内返回）。
如果 Zoekt 能够召回高质量的精准结果，说明用户意图就是确切的代码符号；如果 Zoekt 返回 0 结果（或者评分极低），则立刻降级（Fallback）走 P3 的 `NL Query Rewrite -> 多路召回` 流程。这样可以绝对保证**“P3 自然语言增强永远不会破坏既有的精准检索效果”**。

---

## 2. 多路召回阶段的去重粒度过细 (对应 Section 4.2)

**问题描述：**
在结果融合环节，代码示例中使用了绝对定位 `doc_id = (doc.repo, doc.path, doc.start_line)` 来去重。但在 Zoekt 搜索中，同一个函数或代码流内的多次文本匹配（这些可能由 LLM 生成的多个相近的变体 query 触发），其 `start_line` 可能会有几行到十几行的微小随机偏差。
如果严格按照 `start_line` 去重，同一个文件的同一段逻辑会被当成多条不同的记录项候选，严重挤占 Top 20 以及送给后续 Rerank 模型（只有 25 个）的名额，导致最后推送给用户的搜索结果高度同质化。

**改进建议：使用区间重叠合并（Interval Overlap Merge）**
去重不仅要看绝对的 `doc_id`。引入区间重叠合并逻辑：对于同 `repo` 同 `path` 的匹配，如果两次或多次命中的行号差距在一定窗口范围内（比如上下 15 行以内，或在同一个语法块内），应当融合为单一候选项，并为这个合并项累加多路召回带来的 RRF 分数。

---

## 3. Rerank 模型与上下文缺失的时序矛盾 (对应 Section 1 & 5)

**问题描述：**
在“1. 整体架构”图示和流转时延中，执行时序为 `轻量 Rerank -> 上下文扩展`。这意味着向 Rerank 模型（特别是推荐的 cross-encoder `bge-reranker-v2-m3`）进行文本打分时，拿到的文本可能仅仅是 Zoekt 直接匹配回来的那一行或廖寥几行残缺代码片段。
大语言模型极其依赖充足的代码上下文来判断语义相关度。如果仅输入两三行被截断的代码片段，模型很难打出高分，交叉排序效果会大打折扣。

**改进建议：将特征提权与语义抽样解耦**
要么在发送给 Reranker 之前先做一次中等粒度的上下文补全（如利用已有工具补足完整的 Block块/Method 体）；如果在流程或性能上实在无法前置扩展上下文，那么在 5.1 节提到的 **feature-based rerank**（特征权重算分模型，利用文件名匹配度、符号特征命中情况等直接元数据加权打分）可能比生硬地挂一个黑盒 cross-encoder 更有实效。

---

## 4. RRF 融合机制在小集合场景下的分化不足 (对应 Section 4.2)

**问题描述：**
在 `rrf_merge` 函数中设定了常数因子 `k=60`，但每一条原始通路的 `max_results` 被限制在只有 20。
在典型的网页开源搜索中，K=60 是因为召回列表通常有上百条，能够有极长的长尾。但在这里召回队列极短（Top 20）。在 k=60 的约束下：
* 排名 1 的得分是 `1/61 ≈ 0.0163`
* 排名 20 的得分是 `1/80 = 0.0125`
二者在相加时的差异极其微小。这会导致“即使某一条结果极度精准排名第一，它也会因为竞争对手凭借多条泛指命中积攒的长尾分数而难以拉开差距”。

**改进建议：适配小召回结果集的 K 值调优**
对于极小规模结果集（Top 20）的多路合并，应当对应地将惩罚常数 K 把调到同等量级（例如调整为 `k=10` 到 `k=15`），从而指数级增强处于各个榜单头部的高排名结果在最后融合时的绝对权重优势。

---

## 5. LLM Prompt 的防御性指令不足 (对应 Section 3.2)

**问题描述：**
目前的 Prompt 仅告知模型可以输出 `sym:符号名` 或 `file:路径` 等 Zoekt 格式。然而，大模型在消化用户的全中文意图时，极有可能编造出类似 `{"query": "sym:框架启动功能", "rationale": "..."}` 这种中文与修饰符杂糅的无效伪搜索语句，导致此路查询完全作废。

**改进建议：增强防幻觉边界约束**
Prompt 设定中必须增加一条极其明确的安全限定规则：
**“所有的搜索查询表达式（尤其是 `sym:` 和 `file:` 后引导的内容）只能包含英文字符、数字和代码常用标点（如骆驼拼写法、下划线），绝对不可以包含任何自然语言或中文说明。如果面对无法提炼英文符号的意图需求，仅将提炼出的英文术语作为普通 keyword 输出即可。”**

## 6. 其他需修复的文档错误

*   **标号错乱**：在 `8. 完整请求流程与延迟预算` 之后，紧跟的主标题是 `9. Query API 接口变更`。由于复制排版或者删改过程中的疏漏，其内部标题一下子乱序成了 `8.1 内部接口扩展`、`8.2 返回结果扩展`、最后突变成 `9.3 与 Dify /retrieval 的集成建议`。
*   **正则表达式的作用域**：2.1 节代码中 `re.search(r'[A-Z][a-z]+[A-Z]', query)` 的匹配范畴存在问题。`search` 函数不会做全局或词边界锚定，很容易错误捕获到查询语句里的无效长串中的字母结构。如果不走 Zero-Result Fallback 方案，那么这里就需要一个更为严密精确的词法分类器。
